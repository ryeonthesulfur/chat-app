⚫︎メッセージ投稿機能の実装
　
1. messageモデルの作成 ※コミット名は「messagesテーブルの作成」

ターミナル上で「rails g model message」とコマンド実行する
自動生成されたマイグレーションファイル内に、DB設計のテーブル内容を記述する。

　t.string  :content
　t.references :room, null: false, foreign_key: true
　t.references :user, null: false, foreign_key: true

　上記の意味
テキストの内容：「content」カラム
メッセージの投稿をしたチャットルームのid：「room_id」カラム
メッセージの投稿をしたユーザーのid：「user_id」カラム

【詳細解説】

1. なぜ room_id と user_id が必要なのか？

メッセージアプリにおいて、メッセージは単独では存在できません。
必ず「誰が（User）」「どこで（Room）」発言したかという情報が必要です。

・messages テーブル: メッセージの内容（content）が入っています。
・users テーブル: ユーザー情報が入っています。
・rooms テーブル: チャットルーム情報が入っています。

これらを紐付けるために、messagesテーブルの中に「これはユーザーID:1の人の発言です」、「これはルームID:5での発言です」という名札（ID）をつけておく必要があります。
これが user_id と room_id です。


2. foreign_key: true（外部キー制約）とは？

これはデータベースに設定する**「門番」のような厳しいルール**です。
もしこのルール（制約）がないと、以下のような変なデータが保存できてしまいます。

・存在しないユーザーIDのメッセージ
　　例：ユーザーは5人しかいないのに、「ユーザーID: 9999」からのメッセージが保存されてしまう（幽霊からのメッセージ）。

・存在しないルームIDのメッセージ
　　例：すでに削除されたルームID宛てのメッセージが保存されてしまう（行き先のない迷子のメッセージ）。

foreign_key: true をつけると、データベースが保存の直前にチェックをしてくれます。
「おっと、この user_id: 9999 は users テーブルに実在しないIDですね？ 保存は許可できません！」
と、エラーを出して保存をブロックしてくれます。
これにより、データベースの中身が常に「実在するユーザーとルームに紐付いた正しいデータ」だけに保たれます。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――





2. 投稿したメッセージの情報をDBに保存　　※コミット名は「メッセージ送信機能の実装」

 ・実装に必要なルーティングを設定しましょう

チャットルームへの入り口を作る段階。

Messageモデル（データベース）を作ったので、ルーティングを設定する。

「config/routes.rb」ファイル内にて、

  resources :rooms, only: [:new, :create] do
    resources :messages, only: [:index, :create]
  end

  と記述する。

メッセージを送信するということは、それが送信される空間がなければならない。それがチャットルームである。
どこのチャットルームで送信されるメッセージなのかは、ルーティングで設定できる。そのために、ルーティングネストを行う。
やらないと、以下のようになる。

「メッセージの内容は受け取ったけど、これ、どの部屋（room_id）に表示すればいいの？ 部屋番号が分からないよ！」

ルーティングネストをすることで、まず、「room_messages_path」というパスが自動生成され、
「<%= link_to room.name, room_messages_path(room.id) %>」をクリックすれば、自動で
「/rooms/[:rooms.id]/messages」というURLの形が生成され、クリックしたroom_idが
「[:rooms.id]」の中に入ることで、クリックしたチャットルームのデータベースに紐づけられてるメッセージが表示される。


「app/views/messages/_side_bar.html.erb」ファイル内にて、

      <div class="room-name">
        <%= link_to room.name, room_messages_path(room) %>
      </div>

と記述し、リンク先を作る。

また、ルーティングネストの必要性について、これは選択したチャットルーム内でメッセージを送信するときにも、送信したメッセージがMessageモデル内に保存され、
それがアソシエーションしてるRoomモデル内の該当するチャットルームのデータベースに紐づけられて保存される際にも重要である。

「room_messages_path」は、messages コントローラーのindexメソッドと紐づいており、
そのため、チャットルーム名をクリックすると、
messagesコントローラーのindexアクションが実行され、チャットルームのページに遷移できる。




　・form_withを用いてビューファイルを編集する　※見た目の話ではなく、中身の機能面の話！

送信したメッセージが保存されるようにする。

「app/views/messages/_main_chat.html.erb」ファイル内にて、

<div class="formInput">
  <div class="form">
    <input class="type-message" placeholder="type a message">
    <label class="form-image">
      <span class="image-file">画像</span>
      <input type="file" class="hidden">
    </label>
  </div>
  <input class="send-button" type="submit" value="送信">
</div>

　となっているものを、

<%= form_with model: [@room, @message], html: {class: "form"}, local: true do |f|%>
  <div class="form-input">
    <%= f.text_field :content, class: 'form-message', placeholder: 'type a message' %>
    <label class="form-image">
      <span class="image-file">画像</span>
      <%= f.file_field :image, class: 'hidden' %>
    </label>
  </div>
  <%= f.submit '送信', class: 'form-submit' %>
<% end %>


に変更する。
こうすることで、フォームに入力した内容は、
どこどこのチャットルーム（@room）のなになにというメッセージ（@message）として保存される。

入力したフォームのデータをインスタンスに入れるため、
インスタンス変数をアクション内で定義する。

チャットルームの新規作成について

まず、rooms ビューの index がトップ画面で、これが root_path である。
そして、roomsの index は messages ビューのサイドバー「_side_bar.html.erb」を引き継いでいる。
このサイドバーの新規作成、「new_room_path」をクリックすると、
Prefixは、即座に新規作成画面のためのURLを自動生成し、それをルーティングに投げる。

ルーティングは、rooms の CRUDのメソッド「:new」で対応し、URIパターンと照らし合わせる。
そして、rooms コントローラーの new アクションに投げる。

コントローラーの new アクションは、インスタンス変数「@room」を新規として作り、これを
新規作成用画面のビューファイル「new.html.erb」へ投げる。

ビューファイル内のform_withメソッドにあるmodelオプションが空のインスタンス変数を見て、
フォーム入力が済んだ @room をrooms コントローラーのcreateアクションに渡すことを判断する。
そして、paramsの中のパラメーターがcreateアクションによって呼び出された「room_params」メソッドによって
ストロングパラメーターによる許可を得て、RoomモデルのDBに保存される。
保存されたチャットルームには作成時に id が割り振られている。


次に、作成したチャットルームの表示について

まず、作成した時点のユーザー、すなわち Userモデルのテーブルは、room_userモデルという中間テーブルを介して
Roomモデルのテーブルとアソシエーションの関係にあるので、「<% current_user.rooms.each do |room| %>」
というふうに、Userモデルのdevise「current_user」メソッドが、現在ログインしているユーザーのチャットルームのデータを
一つ一つ id を参照して持ってきてくれる。これをアソシエーションした時点でRailsが自動生成したカラム名によるメソッド、
「name」メソッドによって、「room.name」でチャットルームの名前を取得できる。

この名前に、「room_messages_path(room.id)」をリンクづけることでmessages コントローラーのindexに投げると判断するようにできる。
こうしたリンクは、roomモデルが、messageモデルをアソシエーションで従属させているからできる。

サイドバー（_side_bar.html.erb）のチャットルーム名、
「<%= link_to room.name, room_messages_path(room.id) %>」をクリックした瞬間に、
Prefixの「room_messages_path」がURLの 「/rooms/◯◯/messages」に自動生成で切り替わって、
同時にPrefixがクリックされたidをその「◯◯」の部分に詰めてくれて、ルーティングにリクエストする。

そして、ルーティングはURIパターンの「/rooms/:id/messages」と照らし合わせて、
messages コントローラーのindexアクションに投げようと判断する。

そしてコントローラーのindexアクションは、インスタンス変数「@room」を用意し、
これにRoomモデルから該当するidのチャットルームのデータを引っ張ってきて「params[:room_id]」に詰める。
そしたら、アクション名と同名のindex.html.erbに投げる。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――


入力して model が そのパラメーターを messages コントローラーの create アクションに渡した後、
保存する処理を実装する。
この際、アソシエーション関係をうまく使うことで、
コーディングを短縮でき、ミスタイピングも無くせる。

「親（Room）経由で作れば、自動的に『部屋番号（room_id）』が入った状態で
メッセージ（Message）が作れるよ」 という意味である。

```Ruby
def create
    @room = Room.find(params[:room_id])
    @message = @room.message.new(message_params)
    @message.save
  end



  private

  def message_params
    params.require(:content).permit(:content, :image).merge(user_id: current_user.id)
  end
  ```

・アソシエーション関係を利用しない場合

「どの部屋のメッセージか」を表す room_id を、自分でセットしなければならない。

　　@room = Room.find(1) 　　# 1. まず部屋を見つける

　　@message = Message.new(content: "こんにちは", room_id: @room.id)

# 2. メッセージを作るとき、わざわざ「room_id」を指定しないといけない
# 「このメッセージは、さっき見つけた @room の id (1) ですよ」と教える

これだと、room_id の指定を書き忘れるミスが起きるかもしれません。


・アソシエーション関係を利用する場合

has_many :messages と設定してあるので、Room（親）から直接 Message（子）を生み出せます。

@room = Room.find(1)　　# 1. まず部屋を見つける

@message = @room.messages.new(content: "こんにちは")

# 2. 親（@room）の力を使って、子供（messages）を作る
# Railsが「あ、@roomから作るんだね。じゃあ room_id は勝手に入れておくよ！」とやってくれる




 ・DB保存後のページ遷移を実装しましょう

　if @message.save
      redirect_to room_messages_path(@room)
    else

＊＊redirect_to の仕組み（郵便の例え）＊＊

1. create アクション（手紙を出す）

あなたがフォームにメッセージを書いて「送信」ボタンを押します。
これは、あなたが「この部屋（Room）宛てに、この内容（Message）の手紙をポストに投函する」のと同じです。
@message.save が成功すると、手紙は無事に郵便局（データベース）に届きます。


2. redirect_to（転送シールを貼る）

redirect_to room_messages_path(@room) が実行されます。
これは、郵便局員があなたのブラウザに「この手紙の処理は終わったので、次は『部屋番号1のメッセージ一覧』という住所に行ってください」と書かれた転送シールを貼って送り返すようなものです。
このとき、@room という変数（モノ）を送るのではなく、@room のIDを使って「行き先の住所（URL）」を作っているだけです。


3. index アクション（新しい住所に行く）

転送シールを受け取ったあなたのブラウザは、言われた通り、新しいリクエストとして「『部屋番号1のメッセージ一覧』のページをください！」とサーバーにアクセスし直します。
サーバー側では、messagesコントローラーのindexアクションがゼロから動き出します。
createアクションで使っていた変数（@roomや@message）は、もう存在しません。
indexアクションは、URLにある「部屋番号1」という情報（params[:room_id]）を頼りに、改めてデータベースから部屋の情報や、
その部屋の全メッセージ（さっき保存した新しい手紙も含む）を取り出し、新しいインスタンス変数を作ります。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――



3. メッセージの表示 　※コミット名は「メッセージの表示」


ここまでの実装で、投稿した内容をDBに保存する実装をしました。
続いて、この投稿した内容をそのままアプリケーションで表示するように実装します。

行うことは以下の4つです。

・メッセージ情報を記述するファイルとして、
　　部分テンプレートのmessages/_message.html.erbを新規作成する

・_main_chat.html.erbの仮データである不要な記述を削除して、
　　部分テンプレートの_message.html.erbを読み込む

・コントローラーでメッセージ情報を取得する

・_message.html.erbを編集して、取得したDBの値を表示する

これらを順番に実装します。



まずは、一覧画面で表示するメッセージ情報の取得をする。
messages_controller.rbのindexアクションで@messagesを定義する。

「messages_controller.rb」の それぞれ indexアクションと createアクションの if文のなかに
「@messages = @room.messages.includes(:user)」と記述する。

チャット画面に表示したいのは、メッセージとユーザー情報である。
しかし、メッセージが100件あったら、
100回「このメッセージの投稿者は誰？」とデータベースに問い合わせる処理（N+1問題）が発生してしまいます。 
これを防ぐために、**「メッセージと一緒に『ユーザー情報』をまとめて取ってきて！」**と命令する必要があります。
そのため、「include(:user)」と記述することで、一発で全てのメッセージに紐づいているユーザー情報を持ってきてくれます。


続いて、部分テンプレート化を行います。
今回の部分テンプレートでは、collectionオプションというものを使います。


「app/views/messages/_main_chat.html.erb」ファイル内の「chat-view」の中身を削除する。
代わりに、「<%= render partial: 'message', collection: @messages %>」と記述する。
_message.html.erbを部分テンプレートとして、読み込むことができます。

**「collectionオプション」とは？**

Railsのビュー（View）において、「配列（リスト）の中身を一つずつ取り出して、
同じ部分テンプレートを繰り返し表示したい」 という場面で非常に便利に使える機能です。


1. 何が便利なのか？（Before / After）
通常、配列に入ったデータを表示するには each メソッドを使ってループ処理を書きますが、
collection オプションを使うと、このループを書かずに1行でスッキリ記述できます。

❌ collection を使わない場合（eachで回す）
「@messages から1つ取り出して message に入れ、それを部分テンプレートに渡す」という記述が必要です。

<% @messages.each do |message| %>
  <!-- localsオプションで、変数messageを部分テンプレート内で使えるように渡す -->
  <%= render partial: 'message', locals: { message: message } %>
<% end %>


✅ collection を使う場合
「このコレクション（@messages）の数だけ、部分テンプレートを繰り返してね」と指示するだけで済みます。

<%= render partial: 'message', collection: @messages %>



2. 仕組みとルール
collection オプションを使うと、Railsが以下の処理を自動で行ってくれます。

・自動ループ: 渡された @messages の要素の数だけ、自動的に部分テンプレート（_message.html.erb）を繰り返し描画します。

・自動変数割り当て: 部分テンプレート内では、「部分テンプレート名と同じ名前の変数」 が自動的に使えます。
　　例えば、部分テンプレートが _message.html.erb なら、その中では message という変数に、その回ごとのデータが入ってきます。




3. 実装イメージ

呼び出し元（_main_chat.html.erb など）:
<div class="messages">
  <!-- @messagesの中身を全部表示する。使うテンプレートは _message.html.erb -->
  <%= render partial: 'message', collection: @messages %>
</div>


部分テンプレート（_message.html.erb）: ここでは each を書かなくても、単数形の message 変数が使えます。

<div class="message">
  <div class="upper-message">
    <div class="message-user">
      <!-- message変数には、ループ中の1件分のデータが入っている -->
      <%= message.user.name %>
    </div>
    <div class="message-date">
      <%= message.created_at %>
    </div>
  </div>
  <div class="lower-message">
    <div class="message-content">
      <%= message.content %>
    </div>
  </div>
</div>



まとめ
目的: たくさんのデータを一覧表示するときにコードを短く書くため。
メリット: each 文を書かなくて良いのでビューがスッキリする。処理速度も each よりわずかに高速になる場合がある。
ルール: 部分テンプレート内では、テンプレート名と同じ名前の変数（単数形）でデータを受け取る。

5. 投稿時刻の表示の変更　　※コミット名は「投稿時刻の日本時間への修正」

投稿時刻の表示を日本時刻に変更する実装する。

config の「application.rb」ファイル内にて、

module ChatApp
  class Application < Rails::Application
    # Initialize configuration defaults for originally generated Rails version.
    config.load_defaults 7.1
    config.i18n.default_locale = :ja
    config.time_zone = 'Tokyo'


続いて、config/localesディレクトリに「ja.yml」というファイルを作成し、
以下のような設定を記述します。

config/locales/ja.yml

ja:
  time:
    formats:
      default: "%Y/%m/%d %H:%M:%S"


続いて、この時刻設定を元に、「lメソッド」を利用して、表示する時刻へ反映させます。


l（エル）メソッドは、日付や時刻を表示するRailsのメソッドです。
国や地域によって時刻は異なりますが、lメソッドを使えば指定した現地時間に対応できます。

「app/views/messages/_message.html.erb」ファイル内にて、
    
    <div class="name-and-time">
      <div class="user-name"><%= l message.user.name %></div>
      <div class="time"><%= message.created_at %></div>
    </div>

と「l」を追記する。
lメソッドを用いて記述することで、あらかじめ設定した日本時刻のフォーマットで表示できます。




6. 空のメッセージが送信できないようにする　　※コミット名は「空のメッセージが送信できないバリデーションの設定」

ここまでの実装で、メッセージ投稿機能に関する実装は完了しました。

現状のChatAppは、空の値が送信された場合にそのままテーブルに保存され、一覧に表示されてしまいます。
これは、アプリケーションとして好ましくない仕様です。

最後に、空のメッセージがテーブルに保存されないための実装をしましょう。



　・値が空の場合送信できないようにバリデーションを設定しましょう

「message.rb」ファイル内にて、
「validates :content, presence: true」と記述する。


――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
＊＊レイアウト崩れを防ぐには＊＊

バリデーション設定をしたことで、空のメッセージを送信しても、送られることはなくなるけれども、
バリデーションエラーが発生した際、Railsは親切心から、エラーのあった入力フォーム（今回の場合はメッセージの入力欄）を、
自動的に <div class="field_with_errors">...</div> というHTMLタグで囲みます。


・変更前（正常時）のHTML（簡略版）

<input class="type-message" type="text" ... >


・変更後（バリデーションエラー時）のHTML（簡略版）

<div class="field_with_errors">
  <input class="type-message" type="text" ... >
</div>


この自動で追加されたdivタグが、意図せずCSSの適用を妨げてしまい、
結果として「入力欄が狭くなる」「画像ボタンが左に寄る」といったレイアウト崩れを引き起こしてしまいます。

　＊＊解決策＊＊
このfield_with_errorsが自動で挿入されるのを防ぐ設定をRailsアプリケーションに加えるのが、最も簡単で効果的な解決策です。
「config/application.rb」ファイルに以下の設定を追記することで、この自動挿入を無効化し、レイアウト崩れを防ぐことができます。

config/application.rb
# バリデーションエラー時のfield_with_errorsによるレイアウト崩れを防ぐ
    config.action_view.field_error_proc = Proc.new do |html_tag, instance|
      html_tag.html_safe
    end


解説
この設定は、Railsがエラーフィールドを描画する際に、余計なdivタグを追加せず、
元のHTMLタグ（<input>など）をそのまま表示するように命令するものです。
これにより、バリデーションエラーが発生してもレイアウトが崩れなくなります。

【重要】 configディレクトリ内のファイルを変更した後は、設定を反映させるために、
　　　　　必ずWebサーバー（rails s）を一度停止（Ctrl + C）し、再度起動してください。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――




⚫︎削除機能の実装　　※コミット名は「チャットルームの削除機能の実装」

ここまでの作業で、メッセージ投稿機能の実装は完了しました。
ここからは、前章で作成したチャットルームとそれに紐づくメッセージを削除する機能を実装します。

　・チャット終了機能を実装

親モデル（チャットルーム）を削除すると、子モデル（メッセージ）は迷子になってしまうため、
データベースは削除させてはくれない。
アソシエーション関係において、データベースのデータの整合性は厳格で、これを可能にするために、
「dependent: :destroy 」というオプションを使う。

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
＊＊「dependent オプション」とは＊＊

1. 具体的なイメージ

例えば、「雑談部屋（ID: 1）」というチャットルームがあり、その中に「こんにちは」というメッセージ（room_id: 1）があるとします。

チャットルーム（親）: 雑談部屋 (ID: 1)
メッセージ（子）: 「こんにちは」 (room_id: 1) ← 「ID: 1 の部屋のメッセージだよ」と覚えている
この状態で、親である「雑談部屋」だけを無理やり削除してしまうとどうなるでしょうか？

メッセージ（子）: 「僕は ID: 1 の部屋のメッセージだけど……あれ？ ID: 1 の部屋なんてどこにもないぞ？」
これが**「所属先（参照先）を持たないデータ」、いわゆる「迷子データ」**です。


2. データベースのルール（外部キー制約）
データベース（特に foreign_key: true を設定している場合）は、データの整合性を守るために非常に厳格です。
 「迷子になるようなデータが残ってしまうなら、親データの削除は許可しない！」 とエラーを出して削除を止めてしまいます。
 これが、チャットルームの削除が行えない理由です。


 解決策：dependent: :destroy
この問題を解決するには、Rails側に**「親（部屋）が削除されたら、
道連れにして子（メッセージや中間テーブル）も一緒に削除してね」**という指示を出す必要があります。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――



「room.rb」ファイル内にて、
「has_many :room_users」、「has_many :messages」の後ろに、「dependent: :destroy」と追記する。



削除を行うには、まず、ルーティング設定において、
「resources :rooms」に 「:destroy」を追加する。

それから、rooms コントローラーにて、 delete メソッドを追加する。

  def destroy
    room = Room.find(params[:id])
    room.destroy
    redirect_to root_path
  end


  次にdelete アクションが働く部分テンプレート「app/views/messages/_main_chat.html.erb」ファイル内にて、

  <%= link_to "チャットを終了する", room_path(@room), data: { turbo_method: :delete } %>

  と書き換える。

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
  ＊＊なぜ「room_path(@room)」なのか？＊＊

結論から言うと、**「削除したい部屋の『ID（番号）』をURLに埋め込むため」**です。

1. ルーティングのルール
Railsで resources :rooms を設定すると、削除機能のために以下のようなURLのルール（ルーティング）が自動で作られます。

ルール: DELETE /rooms/:id
意味: 「roomsコントローラーの destroy アクションを実行してね。対象は IDが :id の部屋 だよ」
この :id の部分には、具体的な数字（1 や 5 など）が入らなければなりません。 
もし数字がないと、Railsは**「削除しろと言われたけど、どの部屋を消せばいいの？」**と困ってしまいます。


2. room_path の仕事

Prefix である room_path は、このURLを生成するための便利なメソッド（ヘルパー）です。
リンクをクリックした瞬間に、Rails の Prefix は、自動で「DELETE/rooms/1」のようなURLを生成する。

・もし room_path とだけ書くと……
　　Rails: 「えっ、IDは？ 何番の部屋のパスを作ればいいの？」となり、エラーになります（または意図しないパスになります）。

・room_path(@room) と書くと……
　　Rails: 「@room に入っているのは ID: 1 の部屋データだね。じゃあ、その ID を使って /rooms/1 というURLを作っておくよ！」


　　＊＊まとめ＊＊
「（@room）」が必要な理由は、「削除ボタンを押したときに、『この部屋（ID:〇〇）』を消してね！」という宛先を正確に指定するためです。
