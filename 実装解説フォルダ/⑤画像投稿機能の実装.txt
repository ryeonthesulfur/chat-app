【画像投稿機能の実装】

目的

・Active Storageをインストールすること
・選択した画像を表示すること
・画像の大きさを調節すること
・画像かテキストどちらかが存在している場合は、メッセージの送信を可能とすること


ChatAppの実装では、Active Storageというアップロードツールを使用します。
このActive Storageを用いて、本章では以下のことを行います。

アップロードした画像専用のテーブルを作成する
投稿した画像を表示する
投稿した画像の大きさを調整する

⚫︎画像アップロード機能の実装　※コミット名は「画像表示の実装」

ChatAppに画像をアップロードする機能を実装しましょう。
画像のアップロードには、画像の保存・表示・サイズの調整が必要です。

これらの実装にはまず、RailsのActive Storageという機能を使用します。


  1. 画像加工のツールの導入

　・ImageMagick

コマンドラインから画像に処理を加えることができるツールです。
処理としては、画像の作成やサイズ変更、保存形式の変更などがあります。

ImageMagickはGemではなく、ソフトウェアです。
そのため、Homebrewからインストールします。

   brew install imagemagick



 　・MiniMagick

ImageMagickの機能をRubyで扱えるようにしてくれるGemです。
RailsでImageMagickを扱うために必要となります。

MiniMagickによって、ImageMagickの機能がRailsで使用できるようになります。



 　・ImageProcessing

MiniMagickでは提供できない、画像サイズを調整する機能を提供するGemです。



# Gemfileの一番下に記述する

gem 'mini_magick'
gem 'image_processing', '~> 1.2'

bundle install


「application.rb」ファイル内にて、
「module ChatApp
  class Application < Rails::Application」のなかに、

「config.active_storage.variant_processor = :mini_magick」を追記。


　・Active Storageをインストールしましょう

続いて、Active Storageをアプリケーション内で使用する準備を行います。

　　　rails active_storage:install

rails active_storage:installコマンドを実行すると、
Active Storageに関連したマイグレーションが作成されます。
続けてマイグレートしましょう。

Sequel Aceに、テーブルが３つ追加される。
Active Storageと画像加工ツールの導入は完了です。






  2. 画像の保存

作成したActive Storageのテーブルに、画像を保存するための実装を行います。

行うことは以下の2つです。

Active StorageのテーブルとMessagesテーブルのアソシエーションを定義
messages_controller.rbにて、imageカラムの保存を許可

Messagesテーブルのレコードと画像を紐づけるために「has_one_attached」というメソッドを利用します。

各レコードとファイルを1対1の関係で紐づけるメソッドです。
has_one_attachedメソッドを記述したモデルの各レコードは、それぞれ1つのファイルを添付できます。

class モデル < ApplicationRecord
  has_one_attached :ファイル名
end

:ファイル名には、添付するファイルがわかる名前をつけましょう。
この記述により、モデル.ファイル名で、添付されたファイルにアクセスできるようになります。
また、このファイル名は、そのモデルが紐づいたフォームから送られるパラメーターのキーにもなります。


 ・Messagesテーブルに画像ファイルを紐付けする

MessagesテーブルとActive Storageのテーブルで管理された画像ファイルのアソシエーションを記述します。

app/models/message.rb

class Message < ApplicationRecord
  belongs_to :room
  belongs_to :user
  has_one_attached :image

  validates :content, presence: true
end

このとき、messagesテーブルにカラムを追加する必要はありません。


――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
＊＊「Active Storage」とは＊＊
「Active storage」とは、言うなれば、写真のためのモデル・テーブルのこと。
端的に、「photo モデル」みたいな認識で大枠はオッケー。

1. もし自分で「Photoテーブル」を作るとしたら...
自分で Photo モデルやテーブルを作って管理する場合、以下のようなカラムを自分で設計・追加する必要があります。

filename (ファイル名)
content_type (画像の種類: jpg, pngなど)
file_size (サイズ)
message_id (どのメッセージの画像か)
これを全部自分で作るのは大変です。

Active Storageをインストール(rails active_storage:install)すると、
Railsが自動的にファイル管理専用のテーブル（active_storage_blobs や active_storage_attachments など）を作成してくれます。

これらが**「裏側の Photo テーブル」**として働いてくれます。

・active_storage_blobs: ファイルの名前、サイズ、種類などの情報を保存するテーブル（実質的なPhotoテーブル本体）。
・active_storage_attachments: 「どのメッセージ」と「どの画像(blob)」が紐付いているかを管理する中間テーブル。

　　まとめ
認識: 「写真のためのテーブル（モデル）」という理解でOKです。
メリット: messages テーブルにカラムを追加したり、自分で Photo モデルを作らなくても、has_one_attached :image と書くだけで、
Railsが裏側でよしなに画像を保存・管理してくれます。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――



　・画像の保存を許可するストロングパラメーターにしましょう

imageという名前でアクセスできるようになった画像ファイルの保存を許可する実装を行います。

app/controllers/messages_controller.rb
private

  def message_params
    params.require(:message).permit(:content, :image).merge(user_id: current_user.id)
  end

  このように記述することで、imageという名前で送られてきた画像ファイルの保存を許可できます。




  3. 保存した画像の表示

保存した画像をブラウザ上で表示させるために、ビューファイルに画像を表示させるコードを追記する。


――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
＊＊「image_tag メソッド」とは？＊＊

一言で言うと、**「『この画像を表示して！』とモデルのデータを渡すだけで、
勝手に適切なHTML（<img src="...">）を作ってくれる魔法のメソッド」**です。

1. 何がそんなに便利なの？
通常、Webページで画像を表示するには、HTMLで以下のように画像の「住所（URL）」を正確に指定する必要があります。

<!-- 普通のHTMLの書き方 -->
<img src="https://example.com/images/sample.jpg">


しかし、Active Storageを使って保存した画像は、
Railsの裏側の複雑な仕組みの中（データベースや隠しフォルダの奥深く）に保存されており、
そのURLは人間が手書きできるような単純なものではありません。

もし image_tag を使わずに書こうとすると...
Active Storageの画像のURLは、実際には以下のような非常に長くて複雑なものになります。これを自分で調べて書くのは不可能です。

<!-- ❌ 無理やりHTMLで書こうとした場合（実際にはこんなに複雑！） -->
<img src="/rails/active_storage/blobs/redirect/eyJfcmFpbHMiOnsibWVzc2
        FnZSI6IkJBaHBCZz09IiwiZXhwIjpudWxsLCJwdXIiOiJibG9iX2lkIn19--24e3...">


2. image_tag の魔法
そこで image_tag の出番です。 
このメソッドに、画像が紐付いているモデルのデータ（例: message.image）をポンと渡すだけで、
Railsが**「あ、このデータの画像ですね。URLは裏側で調べておきましたよ」**と、
自動的に正しい <img src="..."> タグを生成してくれます。

✅ Railsでの書き方
<%= image_tag message.image, class: 'message-image' %>

⬇️ 自動生成されるHTML（ブラウザが見るもの）
<img class="message-image" src="/rails/active_storage/blobs/redirect/eyJfcmFpbHMi
   OnsibWVzc2FnZSI6IkJBaHBCZz09IiwiZXhwIjpudWxsLCJwdXIiOiJibG9iX2lkIn19--24e3...">


  ＊＊まとめ＊＊
役割: HTMLの <img> タグを作るメソッド。
「複雑なパスを指定しなくてもいい」の意味:
「画像のURL（パス）」を人間が知らなくても、message.image のように**「モデル名.添付ファイル名」**を渡すだけで、
Railsが勝手にURLを計算して表示してくれる、という意味です。
これにより、開発者は「画像の保存場所」を気にすることなく、
直感的に画像を表示させることができます。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
＊＊「「message.image」とは？＊＊
チャットルームに合計5枚の画像（＝画像付きメッセージが5件）あっても、message.image と書くだけで、
Railsが**「それぞれのメッセージに対応した正しい画像」**を自動で表示してくれます。

なぜそうなるのか、仕組みをわかりやすく解説します。

1. なぜ「message.image」だけで区別できるの？
それは、**「ループ処理（繰り返し）」**をしているからです。

チャット画面を表示するとき、Railsは裏側で以下のような作業を高速で行っています。

・1件目のメッセージを取り出す → message という箱に入れる
　　message.image を実行 → 1件目の画像が表示される

・2件目のメッセージを取り出す → message という箱に入れ直す
　　message.image を実行 → 2件目の画像が表示される

...（これを全メッセージ分繰り返す）
コードで言うと、<%= render partial: 'message', collection: @messages %> の部分がこの「繰り返し」を担当しています。 
このおかげで、同じ message.image という書き方でも、その時処理しているメッセージごとの画像が正しく表示されます。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――


しかし、このままの記述では、画像が存在しない場合でも、画像を表示する記述が読み込まれて、エラーを引き起こしてしまいます。
この問題を解決するために、メッセージに画像が添付されているかをチェックするattached?というメソッドを使用します。


――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
＊＊「attached メソッド」とは？＊＊

attached?メソッドは、Active Storageで管理しているファイルが、
特定のレコード（この場合はメッセージ）に添付されているかどうかを簡単に確認するためのメソッドです。
一言で言うと、**「ファイルが添付されていたら true、されていなかったら false を返す」**という、
有無をチェックするスイッチのような役割を果たします。

なぜ attached? が必要なのか？
チャットのメッセージに画像が添付されている場合と、されていない場合がありますよね。

もし、画像がないメッセージに対して「画像を表示しろ」という命令（<%= image_tag message.image %>）を実行しようとすると、
Railsは「表示する画像がありません！」とエラーを出してしまいます。
attached?メソッドを使うと、if文と組み合わせて「もし画像が添付されていたら、表示する」という条件分岐が書けるため、
このエラーを防ぐことができます。


<%# ... 省略 ... %>

<div class="message">
  <div class="upper-message">
    <div class="message-user">
      <%= message.user.name %>
    </div>
    <div class="message-date">
      <%= l(message.created_at) %>
    </div>
  </div>
  <div class="lower-message">
    <div class="message-content">
      <%= message.content %>
    </div>
    <%# ここからがポイント！ %>
    <% if message.image.attached? %>
      <%= image_tag message.image, class: 'message-image' %>
    <% end %>
  </div>
</div>

<%# ... 省略 ... %>


message.image.attached? の部分で、「messageにimageが添付されていますか？」と確認しています。
true の場合のみ、<%= image_tag message.image ... %> が実行され、画像が表示されます。
false の場合（画像がないメッセージ）は、この部分は無視されるので、エラーが起きません。


　・画像が添付されていることを要素生成の条件にしましょう

app/views/messages/_message.html.erb
<%= image_tag message.image, class: 'message-image' if message.image.attached? %>


「画像の投稿」と「投稿した画像の表示」が可能となりました。


　・テキストと画像を投稿しましょう

現時点の実装では、画像のみの送信はできないため、テキストと一緒に送信してみる。





  4. 画像の大きさの調整

「variant メソッド」

Active Storageを導入している場合に使用可能なメソッドです。
variantメソッドを使用することで、ファイルの表示サイズを指定できます。

モデル.ファイル名.variant(resize: '幅x高さ')

保存されている画像データを指定されたサイズに変換し、その画像を返します。
variantメソッドを使用した画像は、指定された横幅と高さを超えるサイズにはなりません。


　・画像の横幅と高さを指定しましょう

app/views/messages/_message.html.erb
<%= image_tag message.image.variant(resize: '500x500'), class: 'message-image' if message.image.attached? %>




  5. 画像かテキストだけのメッセージの送信　コミット名は「画像のみでもメッセージ送信可能となるバリデーションの設定」

「画像があるときは、テキストが空っぽでも保存していいよ（テキスト必須のルールを免除するよ）」 という設定をする。



validates :content, presence: true, unless: :was_attached?

def was_attached?
  self.image.attached?
end

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
1. validates :content, presence: true
「content（テキスト）は存在しないとダメ（必須）！」という厳しいルールです。
通常なら、テキストが空だとエラーになります。

2. unless
「もし〜〜なら、このルールは無視する（適用しない）」という意味です。
if（もし〜〜ならやる）の逆で、「除外条件」を作ります。

3. :was_attached?
「画像が添付されているか？」を確認するメソッド（下で定義しているもの）を呼び出しています。

つまりつなげると、 「基本はテキスト必須！ でも、画像が添付されているなら、
そのルールは無視してOK（テキスト空でも保存してよし）！」 という意味になります。



 1. なぜ self を使うの？
self は 「今まさにチェックされている、このメッセージ自身」 を指します。

例えば、Aさんが送ったメッセージと、Bさんが送ったメッセージがあったとします。 
プログラムがバリデーション（チェック）をするとき、それぞれのメッセージになりきって考えます。

self.image ＝ 「（今チェック中の）私の画像」
という意味になります。

実は Ruby のルール上、self. を省略して image.attached? と書いても動きます。 
でも、あえて self を書くことで、「他の誰でもない、このメッセージ自身の画像の話をしているんだよ！」 と強調して、
読み間違いを防ぐ意図があります。


2. なぜ「くっついてますか（attached?）」と聞くの？
おっしゃる通り、やりたいことは「画像は添付されていますか？」の確認です。 
ではなぜ attached? という単語を使うかというと、Rails（Active Storage）の仕組みのイメージが関係しています。

Railsの世界では、以下のように管理されています。

メッセージ（本体）: テキストなどが書かれた紙
画像（ファイル）: 写真そのもの

この2つは本来バラバラの存在です。 
それを「クリップや糊でペタッと貼り付ける（Attach）」 ことで、メッセージに画像を持たせています。

Attach（アタッチ） = くっつける、取り付ける、添付する
だから、プログラムへの命令も**「画像は（本体に）くっついていますか？（attached?）」** という聞き方になるんです。


まとめ
コードを日本語訳するとこんなニュアンスになります。

def was_attached?
  # 私(self)の 画像(image) は、 本体にくっついて(attached?) いますか？
  self.image.attached?
end
