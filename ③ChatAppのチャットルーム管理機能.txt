【チャットルーム管理機能の実装】

目的:
ユーザーと1対1でメッセージのやりとりができるチャットルームを作成すること
チャットルーム作成画面で「チャットルーム名の入力」と「ユーザーの選択」をして作成できる機能を実装すること
そのために、「チャットルーム」と「ユーザー」の関連をデータベースでどのように管理するのか理解すること


実装の流れ:
1. チャットルームの新規作成画面を実装する
2. チャットルームの新規作成機能を実装する


0. 事前準備
1. 新規チャットルーム作成画面の実装
2. 新規チャットルーム作成機能の実装



０. 事前準備






1. チャットルームの新規作成画面の実装

　・Roomモデルの作成（roomsテーブルの作成）　※コミット名は「roomsテーブルの作成」
　・画面を表示させよう ※コミット名は「新規作成画面の表示」

・Roomモデルの作成（roomsテーブルの作成） ※コミット名は「roomsテーブルの作成」

チャットルームの情報を管理するRoomモデルを作成します。以下、テーブル設計図。

## rooms テーブル

| Column | Type   | Options     |

| name   | string | null: false |


ターミナルにて、「rails g model room」を実行。
マイグレーションファイル内に、
「t.string :name, null: false」を追記し、マイグレイトを実行。
　→ DBのカラムに「name」が追加される。

2. チャットルームの新規作成機能を実装する  

　・中間テーブルの作成　※コミット名は「room_usersテーブルの作成」

中間テーブルを作成するためのには、RoomUserモデルを作成し、
roomsテーブルとusersテーブルの関連を定義します。
こうすることで、一人のユーザーが複数のチャットルームに参加することができ、
且つ、一つのチャットルームに複数のユーザーが所属できるようになる。

ターミナル上で、「rails g model room_user」を実行し、外部キーのカラムを追加する。
マイグレーションファイル内のchangeメソッドにて、

      t.references :room, null: false, foreign_key: true
      t.references :user, null: false, foreign_key: true

の二つを追記する。マイグレイト後、ローカルサーバーを再起動し、アソシエーションを定義します。

---------------------------------------------------------------------------------------------
「t.references :カラム名」
ポイントは、t.references :userと記載すれば、「user_id」というカラムが生成される点です。
また、foreign_key: trueという記述を加えることで、外部キー制約を設定できます。
---------------------------------------------------------------------------------------------

多対多のアソシエーションを記述する。

「app/models/room.rb」ファイル内に

　　has_many :room_users
　　has_many :users, through: :room_users

と追記する。
　

「app/models/user.rb」ファイル内に

　　has_many :room_users
　　has_many :rooms, through: :room_users

と追記する。


「app/models/room_user.rb」ファイル内に

　　belongs_to :room
　　belongs_to :user

と追記する。



　・コンソールでデータを追加

「room_users」モデルと中間テーブルを作成したことで、コンソールからDBにデータを保存できるようになった。
「rails c」でターミナル（コンソール）を開き、「user_ids」という特殊なキーで以下、データを保存する記述をする。

　　[1] irb(main):002:0> Room.create(name: "ルーム１", user_ids:[1, 2])

こうすることで、roomsテーブルに「id1」のチャットルームが作成され、
中間テーブルにも同様に、「room_id」ができる。

「user_ids」というキー名がポイント。
roomという親要素一つを保存するときに、「モデル名」＋「_ids」というキーを使用すると、
例えば、「user_ids」で所属させたいユーザーを配列で指定した場合、その情報が中間テーブルに保存される仕組みとなっている。

roomsテーブルには、「[id: 1, name: ルーム1]」の情報が、
room_usersテーブルには、「[id: 1, room_id: 1, user_id: 1], [id: 2, room_id: 1, user_id: 2]」
といったふうに、それぞれの情報が保存される。

この方法を利用してチャットルーム作成機能を実装する。




　・チャットルームに所属させるユーザー情報を送信しよう　※コミット名は「プルダウンの実装」

新規チャットルーム作成画面で、チャットルームに所属させるメンバーをユーザーの中から選択する、プルダウンを実装する。
まずは、このプルダウンに仮データを表示できるよう実装する。
その後、DBに保存されているユーザーの情報を取得して、プルダウンに表示する実装を行う。

プルダウンは、select要素とoption要素を使用して実装します。
これらのHTMLの要素を使うことで、プルダウンと選択項目を実装できます。


――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
**select要素とは**
select要素は、プルダウンのようなセレクトボックスを作成するための要素のこと。
select要素には、name属性のような任意の値を指定することで、
paramsを通して、その値（データ）をコントローラーへ渡せる。


****option要素とは**
option要素は、select要素の中に配置される選択肢を定義するための要素。
option要素には、value属性のような任意の値を指定することで、選択された値をコントローラーへ渡せる。
――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――

「app/views/rooms/new.html.erb」ファイル内にて、

　　<select>
    　<option>チャットするユーザーを選択してください</option>
      <option>ユーザー1</option>
      <option>ユーザー2</option>
  　</select>

  と記述する。
  こうすることで、プルダウンにそれぞれ「ユーザー１」、「ユーザー２」という選択肢が表示される。
  この仮の選択肢から、実際のDBからの情報を反映させた選択肢に変更する。




　・DB上のユーザーの情報を取得しましょう

ビューファイル上に、DBに管理されているユーザーすべてのレコードをUserクラスから取得して、
それぞれ1つずつユーザー名を表示させる実装を行います。

「app/views/rooms/new.html.erb」ファイル内にて、

　　　<select>
        <option value="">チャットするユーザーを選択してください</option>
        <% User.all.each do |user| %>
          <option><%= user.name %></option>
        <% end %>
    　</select>

と記述内容を変更する。
こうすることで、DBに存在するユーザー名を表示させることができた。
しかし、すでにログインしているユーザーも選択肢に含まれているため、
これを除くようにする。
チャットルームを作成するユーザー本人は、選択せずともルームのメンバーに含める方が理にかなっているからである。

※「<option value="">チャットするユーザーを選択してください</option>」の
「value=""」に関しては、ここに「 ""（空文字）」を指定しておくことで、
ユーザーがこの選択肢を選んだまま送信ボタンを押したときに、
「データは空です（誰も選んでいません）」 という情報を送ることができます。



　・選択肢から現在ログインしているユーザー名を取り除く

「app/views/rooms/new.html.erb」ファイル内にて、

　　　<select>
          <option value="">チャットするユーザーを選択してください</option>
          <% User.where.not(id: current_user.id).each do |user| %>
            <option><%= user.name %></option>
          <% end %>
     </select>


「モデル名.where("条件")」と記述すると、条件に一致したレコードを配列として取得できる。
このwhereメソッドに続けて「notメソッド」を追記すると、「モデル名.where.not("条件")」は、
条件に一致したレコード以外の値を配列として取得できる。

以上で、プルダウンの表示に関する実装が完了した。
続いて、このフォームからパラメーターとしてコントローラーへ送る値を指定する。




　・選択肢したユーザーの情報をコントローラーへ送る為に、paramsに値（パラメーター）を指定する。

「app/views/rooms/new.html.erb」ファイル内にて、

<select name="room[user_ids][]">
  <option value="">チャットするユーザーを選択してください</option>
  <% User.where.not(id: current_user.id).each do |user| %>
    <option value=<%= user.id %>><%= user.name %></option>
  <% end %>
</select>


「<select name="room[user_ids][]”>」の記述により、コントローラー側では params という箱の中に、以下のような構造でデータが届きます。

params = {
  room: {
    user_ids: ["選択されたユーザーのID"]
  }
}

これは、「room」という大きな箱の中に、「user_ids」という小さな箱が入っている状態である。
もし「入れ子」にせず、バラバラに送ってしまうと……

# バラバラの場合
params = {
  "room_name" => "雑談部屋",
  "user_ids" => ["1", "2"]
}

コントローラーで保存するときに、「えーっと、名前はこれと、ユーザーはこれと……」と一つずつ拾い集めないといけません。
しかし、「入れ子（room グループ）」にしておけば：

# 入れ子の場合
params = {
  "room" => {
    "name" => "雑談部屋",
    "user_ids" => ["1", "2"]
  }
}

コントローラーでは、「room の箱に入ってるやつ、全部まとめて保存して！」 （Room.create(room_params)）と一言で命令できるようになります。
つまり、room[user_ids] という書き方は、「これは room グループのデータだよ！」と名札をつけて、Railsに整理整頓をお願いしている と考えるとわかりやすいです。

しかし、現状の実装のままでは、現在ログインしているユーザーのidを送信することができず、
送信したユーザーのみチャットルームに存在していることになってしまいます。
したがって、現在ログイン中のユーザーのidも一緒に、paramsでコントローラーへ送信できるよう実装します。


　・パラメーターにログイン中ユーザーのidも含める


　　　<select name="room[user_ids][]">
  　　　<option value="">チャットするユーザーを選択してください</option>
  　　　<% User.where.not(id: current_user.id).each do |user| %>
    　　　<option value=<%=user.id%>><%= user.name %></option>
  　　　<% end %>
　　　</select>
　　　<input name="room[user_ids][]" type="hidden" value=<%= current_user.id %>>


《上記コードの解説》
1. やりたいこと

チャットルームを作るとき、画面のプルダウンで選ぶのは「チャットする相手」だけですよね。 
でも、データベースには**「相手」と「自分（作成者）」の2人が参加する**と記録しないといけません。
わざわざプルダウンで「自分」を選ばせるのは面倒なので、
「画面には出さないけど、裏側でこっそり自分のIDも一緒に送っちゃおう」 というのがこのコードの目的です。


2. コードの仕組み

・type="hidden" （隠しポケット） 画面上には何も表示されません。
　　ユーザーには見えない「隠しポケット」を作ります。

・value=<%= current_user.id %> （中身は自分） その隠しポケットの中に、
　「ログインしている自分のID」をあらかじめ入れておきます。

・name="room[user_ids][]" （同じ宛名） ここが最大のポイントです。
　 さっきのプルダウン（相手を選ぶ場所）と全く同じ名前をつけています。


3. データが合体して送られる！

「相手を選ぶプルダウン」と「自分の隠しポケット」に同じ名前がついているため、
送信ボタンを押すと、Railsがこれらを1つのリスト（配列）に合体させてくれます。

・プルダウンで選んだ値： 2 （相手）
・隠しポケットの値： 1 （自分）
     ↓ 送信！

コントローラーに届くデータ：
params = {
  "room" => {
    "user_ids" => ["2", "1"]  # 相手と自分が合体して届く！
  }
}


**まとめ**
この記述のおかげで、コントローラー側で「自分を追加する処理」などをわざわざ書かなくても、
「送られてきたIDをそのまま保存するだけで、自分も相手もメンバー登録完了！」 という便利な状態を作ることができます。


――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
**「<select name="room[user_ids][]”>」の意味**

フォームが送信されたときに 「選択されたユーザーのIDを、配列（リスト）としてサーバーに送る」 ための記述です。

1. name 属性の役割（荷札の宛名）

役割: サーバーにデータを送るとき、「このデータはどの箱（パラメータ）に入れるか」 を指定するラベルです。
このコードの場合:


2. room[...]

「room」は、Room モデルのデータだよ、という大枠。
データを room というグループにまとめます。
コントローラー側では params[:room] でデータを受け取れるようになります。


3. user_ids

「user_ids」は、Room モデルの「ユーザーIDリスト」だよ、という指定。
Room モデルが持つ「ユーザーとの関連付け（has_many :users）」を更新するための特別な名前です。
これは Railsの特別な命名ルール です。
Room モデルは has_many :users（多対多）の関係を持っているため、user_ids という名前でIDのリストを渡して保存すると、
Railsが自動的に中間テーブル（room_users）にもデータを保存してくれます。
つまり、「このルームには、このユーザーIDの人たちが参加します」という情報を一括で保存するための仕組みです。


4. []

「[]」は、複数の値が入るかもしれないから「配列（リスト）」で送るよ、という記号。
データを「配列（リスト）」として送るための記号です（これがないと、RailsがIDをうまく受け取れない場合があります）。
データを 「配列（複数の値が入る箱）」 として送るという意味です。
これがないと最後の1つの値しか送られませんが、これがあることで「選択されたID」をリスト形式でコントローラーに渡すことができます。


　**まとめ**
これがあるおかげで、コントローラー側（rooms_controller.rb）は 
「params[:room][:user_ids]」と書くだけで、送られてきたデータを受け取ることができます。
もし name がないと、サーバーはデータを受け取れません。
また、コントローラー側で特別な処理を書かなくても、
「ルームを作成すると同時に、選んだユーザーをそのルームのメンバーとして登録する」 ことが可能になります。

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
**value 属性の役割**

value は、フォームに入力されたデータを 
「サーバー（コントローラー）に正しく届けるための中身」 の役割をしています。
以下、詳しく解説します。

　・value 属性の役割（荷物の中身）
<option value=<%= user.id %>> の部分です。

役割: ユーザーがその選択肢を選んだときに、「実際にサーバーに送信されるデータ」 です。
なぜ必要なのか？:
画面に見えているもの: <%= user.name %>（例：「田中太郎」）
システムに必要なもの: <%= user.id %>（例：「1」）
データベースに保存するときは、「田中太郎」という文字ではなく、「IDが1番のユーザー」として保存する必要があります。そのため、画面には名前を表示しつつ、裏では value に設定された ID を送信しています。
具体的なデータの流れ
例えば、画面で「田中太郎（ID: 1）」を選んで送信ボタンを押すと、以下のようなデータがサーバーに飛んでいきます。

ruby
# コントローラーに届くデータ (params)
{
  "room" => {
    "user_ids" => ["1"]  # nameで指定した場所に、valueの値が入る
  }
}

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――
**collection_select メソッドの役割**

collection_selectメソッドとは、データベースから取得した複数のレコード（今回はUser）から選択肢を自動生成するメソッドです。

現在、<select> タグをHTMLで直接書いていますが、
Railsのヘルパーメソッド collection_select を使うと、よりスッキリと記述できます。


    <%= f.collection_select :user_ids, User.where.not(id: current_user.id), 
      :id, :name, { include_blank: 'チャットするユーザーを選択してください' }, 
      { name: 'room[user_ids][]' } %>


・「:user_ids」 これを指定すると、自動的に name="room[user_ids][]" を作ってくれます。
・「:id」 これが value （送信される値）になります。
・「:name」 これが画面に表示される文字になります。

・「:user_ids」 を指定することで、モデルとの連携が明確になります。
・「User.where.not(...)」 で自分以外のユーザーを取得しています。
・「:id」 が送信される値（value）、:name が表示される文字です。

最後の { name: 'room[user_ids][]' } で、送信されるパラメータ名を明示的に指定しています
（これがないと単一の値として送られ、保存に失敗する可能性があるため）。

――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――――

"room"=>{"name"=>"新規チャット", "user_ids"=>["1", "2"]} の「"room"」が @room のキー。
form_withのオプションに @room を指定したことで、このフォームから送られるユーザーデータがパラメーターとなって、
roomキーの値（ハッシュ）に入る。




　・ DBに値を保存できるようにコントローラーを編集する

Railsではセキュリティ対策として、保存する前にストロングパラメーターを使って許可を出す。
今回送られてくるパラメーターは配列になっていて、これに対し、
配列として受け取ったデータを含むストロングパラメーターを定義します。
その後、値が保存できているか実践して確認する。


《app/controllers/rooms_controller.rb》
```Ruby
  def create
    @room = Room.new(room_params)
    if @room.save
      redirect_to root_path
    else
      render :new, status: :unprocessable_entity
    end
  end


   private

  def room_params
    params.require(:room).permit(:name, user_ids: [])
  end
```

保存が成功したかどうかで処理を分岐しています。
成功した場合はredirect_toメソッドでルートパスにリダイレクトし、
失敗した場合はrenderメソッドでrooms/new.html.erbのページを表示しています。




　・バリデーションの設定

「app/models/room.rb」ファイル内にて、
「validates :name, presence: true」を追記する。
